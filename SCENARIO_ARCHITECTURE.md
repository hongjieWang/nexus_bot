# 核心交易场景架构与业务规则 (Scenario Architecture & Business Rules)

本文档旨在明确本项目的两个核心高频交易场景的业务规则、技术选型及执行链路。所有系统重构与代码实现均必须严格遵循本文档确立的原则。

## 架构定调：Go 执行，Python 研究
*   **高频事件驱动与执行 (场景A、B)**：100% 采用 Go (Golang) 实现。追求极致低延迟，禁止在关键路径中引入跨语言 IPC 通信。
*   **中低频统计与 AI 预测 (未来规划)**：采用 Python 实现，通过异步消息系统向 Go 发送交易信号。

---

## 场景 A：Binance Alpha (币安上币/早期项目发现)

### 1. 业务本质
一个极度内卷的**“消息驱动”**抢跑游戏。核心目标是在币安发布上币公告或新增资产配置的极短时间内，在 BSC/链上 DEX (如 PancakeSwap) 抢先买入该代币。

### 2. 信号源提取策略 (Binance Private WebSocket API)
*   **方案**：彻底放弃公共 HTTP 轮询爬虫，改用**币安官方私有 WebSocket 数据流通道 (`wss://api.binance.com/sapi/wss`)**。
*   **连接鉴权**：
    *   **签名规则**：使用 `API_SECRET` 对 URL 参数 (`random`, `topic`, `recvWindow`, `timestamp`) 进行 HMAC SHA256 哈希生成 `signature`。
    *   **Header 注入**：在 WSS 握手请求头中必须注入 `X-MBX-APIKEY` 以验证身份。
*   **连接保活机制 (心跳/风控)**：
    *   **主动 PING**：客户端必须每隔不超过 30 秒 (建议 25 秒) 发送一次空的 PING 帧。若 1 分钟未收到，服务器将强制断开。
    *   **限流控制**：严格遵守每秒最多 5 条消息的限速 (包含 PING/PONG 和 JSON 负载)，防止 IP 被封禁。
    *   **有效期**：每个连接最长存活 24 小时，业务逻辑层必须实现自动重连与订阅状态恢复。

### 3. 交易执行策略 (Sniper Execution)
由于上币瞬间价格剧烈波动，必须采取激进但可控的执行策略：
*   **滑点 (Slippage) 推荐**：**20% ~ 30%** (极高滑点以确保交易打包成功，对抗瞬时拉升)。
*   **单笔风控**：严格限制单笔买入金额 (例如固定 0.1 BNB)，严禁 All-in。
*   **Gas 策略 (防夹/抢跑)**：采用高 Gas Tip (贿赂矿工) 策略，显著提高 Gas Price (如 3~5 倍)，争取 Block 0/1 确认。
*   **执行场所**：直接调用目标链 (BSC) 的 DEX Router (如 PancakeSwap) 发起市价买入。

## 场景 B：Meme Rush (BSC 链上仿盘打新)

### 1. 业务本质
一个基于**“链上合约事件驱动”**的扫雷与高死亡率 (PvP) 游戏。监听类似 Four.Meme 的发币工厂 (Factory)，在极早期介入。

### 2. 信号监听与入场条件 (三关校验)
监听到 `TokenCreated` 事件后，**不盲目冲入**，必须在极短时间 (<1s) 内串行通过以下校验：
*   **第一关 (防貔貅/高税)**：调用安全 API (如 GoPlus)。买入/卖出税必须 `< 10%`，且绝不能是 Honeypot。
*   **第二关 (开发者筹码)**：校验部署者 (Dev) 钱包余额，若预留代币比例过高 (例如 `> 5%`)，直接放弃 (防砸盘跑路)。
*   **第三关 (聪明钱跟车)**：利用本地 Alpha Engine 校验开盘前几个区块内，是否有已监控的“高分聪明钱”地址买入。有则跟随。

### 3. 仓位生命周期管理 (Lifecycle & Exit Strategy)
Meme 币波动极大，必须实行铁血的纪律性卖出策略：
*   **建仓**：校验通过后立即市价买入，并在内存中生成 `Position` (持仓) 对象。
*   **巡逻轮询**：高频 (如每 3 秒) 查询该池子的实时价格。
*   **自动止盈 (TP - Take Profit)**：**铁律：涨幅达到 +50%**。触发后无情调用合约 (如 Four.Meme 的 `sell` 或 DEX Router) 市价砸盘清仓。
*   **自动止损 (SL - Stop Loss)**：**建议：跌幅达到 -30%**。Meme 币洗盘剧烈，止损设置稍宽，但一旦触达必须认赔市价割肉。
*   **极端逃生 (Rug Panic)**：若监听到池子流动性骤减 (撤池子前兆)，无视当前价格与设定滑点，采用 100% 滑点发起最高优先级逃生交易。

## 场景 C：智能趋势预测 (K线 + 资讯情绪多空信号) - 后期规划

### 1. 业务本质
一个基于**“统计学与机器学习”**的中低频趋势交易系统 (偏向合约与现货波段)。通过融合技术面 (K线指标) 与基本面 (自然语言情绪)，在震荡或趋势行情中输出高置信度的做多 (LONG) 或做空 (SHORT) 信号。

### 2. 混合架构分工 (Python 大脑 + Go 手脚)
此场景完美契合**“Python 负责研究与预测，Go 负责风控与执行”**的行业标准微服务架构。

*   **Python 层 (策略研发与 AI 引擎)**：
    *   **数据采集**：异步拉取交易所 K 线历史数据 (如 15m, 1H)，同时通过 Twitter/币安广场/新闻 API 获取实时资讯文本。
    *   **特征提取**：利用 `ta-lib` 和 `pandas` 极速计算 MACD、RSI 等技术因子。
    *   **NLP 情绪打分**：调用大语言模型 (LLM) 或本地预训练 NLP 模型，将资讯文本转换为量化的情绪得分 (Sentiment Score)。
    *   **信号生成**：综合技术面和情绪面，通过机器学习模型 (如 XGBoost/LightGBM) 或固定权重逻辑，生成结构化的交易信号 (包含方向、胜率置信度、建议止盈止损位)。
    *   **内部通信**：将最终信号 (如 `{"symbol": "BNBUSDT", "action": "LONG", "confidence": 0.88}`) 经由内网高性能消息总线 (如 Redis Pub/Sub, ZeroMQ) 异步推送给 Go 引擎。

*   **Go 层 (风控拦截与极速执行)**：
    *   **信号消费**：独立 Goroutine 实时监听 Python 推送过来的多空信号。
    *   **风控校验 (Risk Layer)**：在发单前，强制校验账户可用可用保证金 (Margin)、单币最大风险暴露 (Exposure)、系统连续亏损熔断等刚性风控规则。
    *   **交易执行**：风控通过后，安全地调用币安合约 (Binance Futures) API 完成极速下单。

### 3. 核心优势
*   **无惧通信延迟**：此类波段策略主要关注分钟级以上的趋势，Python 计算和跨语言通信带来的几十毫秒延迟对策略收益毫无影响，从根本上避开了跨语言 IPC 的缺点。
*   **生态降维打击**：直接复用 Python 在数据科学、机器学习及历史回测 (Backtesting) 领域的无敌生态，无需在 Go 中痛苦地手撸数学公式与指标。
*   **安全隔离**：即使 Python 策略引擎因为内存泄漏或死循环崩溃，底层的 Go 执行引擎与风控层依然坚如磐石，能继续守卫当前持仓并正常执行止损逻辑。
