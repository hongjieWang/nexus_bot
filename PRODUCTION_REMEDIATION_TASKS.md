# 生产可用性整改任务清单

本文档用于承接本项目最新一轮“生产可用性 Review”的整改工作，将发现的问题转成可执行任务。

适用范围：
- Go 后端交易引擎
- DEX Sniper
- Dashboard API
- React Dashboard
- 运行与部署配置

执行原则：
- 先修复会导致资金风险、配置被篡改、监控失效的问题
- 再修复会导致运行不一致、部署困难、可维护性差的问题
- 每项任务必须有明确的验收标准

---

## P0 必须优先完成

### [ ] T001 为 `/api/config` 增加鉴权与最小权限控制

**问题**
- 当前 `/api/config` 没有任何鉴权。
- CORS 允许任意来源访问。
- 服务端会直接持久化调用方提交的任意 key。

**整改目标**
- 只有受信任的前端或运维调用方可以修改配置。
- 服务端只允许更新白名单内的配置项。
- 禁止任意来源跨域写配置。

**任务内容**
- 为 `/api/config` 增加鉴权机制。
- 增加配置项白名单，例如：
  - `DEX_SNIPER_ENABLED`
  - `TRADE_ENABLED`
  - `EXECUTION_PRIVATE_KEY`
  - `BINANCE_API_KEY`
  - `BINANCE_API_SECRET`
  - `TRADE_USDT`
- 将 `Access-Control-Allow-Origin: *` 改为可配置的受信任 Origin 列表。
- 对未授权请求返回 `401/403`。
- 对非白名单字段返回 `400`。
- 增加审计日志，记录配置修改时间、来源和字段名，不记录明文值。

**涉及文件**
- `api/api.go`
- `config/config_manager.go`

**验收标准**
- 未携带凭证的请求不能读写敏感配置。
- 非白名单字段无法写入数据库。
- 非受信任 Origin 的浏览器请求被拒绝。
- 接口错误码与错误消息明确。

---

### [ ] T002 禁止使用默认主密钥启动

**问题**
- `MASTER_KEY` 缺失或长度不足时，系统会回退到固定默认密钥。
- 这会导致所有加密数据在默认密钥已知的情况下可被解密。

**整改目标**
- 生产环境中不存在“兜底默认密钥”。
- 密钥不合法时系统必须拒绝启动。

**任务内容**
- 删除 `default_master_key_must_be_32_bt` 回退逻辑。
- 启动时校验 `MASTER_KEY` 是否存在且长度满足要求。
- 对不合法配置直接终止启动，并输出明确错误日志。
- 补充 README 或部署文档中的配置要求。

**涉及文件**
- `config/config_manager.go`
- `README.md`

**验收标准**
- 未配置 `MASTER_KEY` 时后端启动失败。
- `MASTER_KEY` 长度不满足要求时后端启动失败。
- 正常密钥配置下，加解密读写行为保持可用。

## P1 高优先级整改

### [ ] T003 修复配置保存成功但实际失败的问题

**问题**
- API 层没有检查 `config.SetConfig` 的返回值。
- 数据库未初始化或写入失败时，前端仍可能收到成功响应。

**整改目标**
- 配置写入结果必须真实反映后端执行结果。

**任务内容**
- 在 `/api/config` 的 POST 逻辑中逐项检查 `SetConfig` 返回值。
- 一旦出现写入失败，返回明确错误码和错误消息。
- 对多字段写入采用“全部成功才返回成功”的策略。
- 视情况增加事务，避免部分字段写入成功、部分失败。

**涉及文件**
- `api/api.go`
- `config/config_manager.go`

**验收标准**
- 数据库不可用时，接口返回失败而不是成功。
- 任意字段写入失败时，调用方可以感知失败原因。
- 前端错误提示与真实状态一致。

---

### [ ] T004 修复 DEX 卖出失败后持仓状态丢失的问题

**问题**
- `executeSell` 在卖单真正成功前就把持仓从内存中删除。
- 如果广播失败或链上回滚，系统会失去对该仓位的跟踪能力。

**整改目标**
- 卖出流程要么成功关闭仓位，要么保留/恢复仓位继续监控。

**任务内容**
- 将 `positions` 中的仓位状态扩展为：
  - `open`
  - `selling`
  - `sell_failed`
- 卖出广播前不要直接删除仓位。
- 卖出失败时恢复为可监控状态，支持重试或人工介入。
- 卖出成功后再正式移除仓位。
- 为失败卖出增加告警日志。

**涉及文件**
- `dex_sniper.go`

**验收标准**
- 卖出广播失败后，仓位不会凭空消失。
- 回执失败后，仓位仍可继续被监控。
- 卖出成功后，仓位才会从内存状态中移除。

---

### [ ] T005 修复 `monitorRugPulls` 对 `nil` 持仓的崩溃风险

**问题**
- Sniper 买入时会先写入 `nil` 占位。
- `monitorRugPulls` 遍历 `positions` 时直接访问 `pos.PoolAddress`，可能触发空指针崩溃。

**整改目标**
- 任何监控循环都必须对临时态、空态、异常态持仓安全处理。

**任务内容**
- 在 `monitorRugPulls` 构造 `poolAddresses` 时过滤 `nil`。
- 统一封装“获取可监控持仓快照”的辅助方法。
- 为长生命周期 goroutine 增加 `recover` 和错误日志，避免单个 bug 导致整个监控线程退出。

**涉及文件**
- `dex_sniper.go`

**验收标准**
- 存在 `nil` 占位仓位时不会 panic。
- Rug pull watcher 异常后可记录日志并继续运行。

---

### [ ] T006 补齐实时下单与撤单失败的处理闭环

**问题**
- `placeOrder`、`cancelAllOrders` 的错误在策略执行主流程中被忽略。
- 这会导致系统“以为已经挂上保护单”，但交易所实际上可能拒绝了请求。

**整改目标**
- 下单失败、撤单失败、保护单创建失败都必须成为显式状态，而不是静默吞掉。

**任务内容**
- 在 `executeDecisions` 中检查：
  - 开仓单结果
  - 撤单结果
  - 止损单结果
  - 止盈单结果
- 若保护单下单失败：
  - 立即记录高优先级日志
  - 触发告警
  - 可选：暂停该策略实例继续开新仓
- 对 Binance 返回的错误内容做结构化日志输出。

**涉及文件**
- `trading.go`

**验收标准**
- 任一交易所请求失败时日志中有明确的动作、参数和错误原因。
- 保护单下单失败时系统不会继续无保护裸奔。
- 策略在关键错误下具备停机或降级能力。

## P2 中优先级整改

### [ ] T007 加固 `fetchKlines` 的 HTTP 与数据解析安全性

**问题**
- 当前代码没有检查 HTTP 状态码。
- 使用了未保护的下标访问和类型断言。
- 一旦上游返回异常 payload，可能直接 panic。

**整改目标**
- 外部行情接口异常时，系统应安全失败而不是崩溃。

**任务内容**
- 检查 HTTP 状态码，非 `2xx` 直接返回错误。
- 校验响应 JSON 是否符合预期结构。
- 去除危险的裸类型断言与越界访问。
- 对解析错误打日志并让当前轮次安全退出。

**涉及文件**
- `trading.go`

**验收标准**
- 行情接口返回错误 JSON 时不会 panic。
- HTTP 4xx/5xx 时能够正常返回错误。
- 非法 K 线数据不会污染策略执行。

### [ ] T008 让前端 API 地址可配置，支持真实部署

**问题**
- Dashboard 当前写死 `http://localhost:18080/api`。
- 在反向代理、容器部署、远程环境中不可直接使用。

**整改目标**
- Dashboard 可以根据环境变量或相对路径接入后端。

**任务内容**
- 将 `API_BASE` 改为从 Vite 环境变量读取。
- 提供默认值，例如相对路径 `/api`。
- 补充 `.env.example` 或 dashboard README。

**涉及文件**
- `dashboard/src/App.tsx`
- `dashboard/README.md`

**验收标准**
- 本地开发可继续使用。
- 通过环境变量可以切换到其他 API 地址。
- 反向代理部署时无需改源码。

---

### [ ] T009 为关键路径补充最小自动化测试

**问题**
- 当前仓库中没有自动化测试文件。
- 交易、配置、风控、数据解析缺少回归保障。

**整改目标**
- 至少为最关键的高风险逻辑建立最小回归网。

**任务内容**
- 为以下模块补测试：
  - `config/config_manager.go`
  - `api/api.go`
  - `trading.go` 中的关键纯逻辑
  - `strategy_loader.go`
  - `dex_sniper.go` 中的状态流转逻辑
- 优先增加：
  - 配置保存失败测试
  - `MASTER_KEY` 校验测试
  - `fetchKlines` 异常响应测试
  - `executeSell` 失败恢复测试
  - `monitorRugPulls` 的 `nil` 仓位测试

**验收标准**
- 新增测试可在 CI 或本地稳定执行。
- 至少覆盖本轮 Review 的高风险回归点。

---

### [ ] T010 补齐 DEX 卖出后的数据库落账

**问题**
- DEX 卖出成功后仍有 `TODO: 记录到数据库`。
- 这会导致 Dashboard、复盘、PnL 统计与真实链上行为不一致。

**整改目标**
- DEX 开仓与平仓结果都能被可靠记录。

**任务内容**
- 为 DEX 持仓建立完整交易落库逻辑。
- 卖出成功后记录：
  - token
  - side
  - 数量
  - 入场价格
  - 出场价格
  - PnL
  - 触发原因（TP/SL/Panic）
  - 时间戳
- 补充与 Dashboard 指标的一致性检查。

**涉及文件**
- `dex_sniper.go`
- `database/db.go`
- `api/api.go`

**验收标准**
- DEX 成交后 `TradeHistory` 或新增 DEX 表中有完整记录。
- Dashboard 指标能反映真实 DEX 成交结果。

---

### [ ] T011 为长生命周期 worker 增加统一的异常恢复与告警

**问题**
- 多个 goroutine 长时间运行，但没有统一的 `recover`、退出重启、健康状态暴露机制。

**整改目标**
- 任意单个 worker 异常退出后，系统可观测、可恢复。

**任务内容**
- 为以下 worker 增加统一保护层：
  - Alpha Engine 周期任务
  - V2/V3 订阅循环
  - Sniper `monitorPositions`
  - Sniper `monitorRugPulls`
  - Trading Engine 周期循环
- 增加 panic recover 日志。
- 暴露基础健康状态到 `/metrics` 或新的健康检查接口。

**涉及文件**
- `main.go`
- `alpha_engine.go`
- `dex_sniper.go`
- `trading.go`

**验收标准**
- worker panic 时进程不会无日志静默失效。
- 至少能从日志或 metrics 看出哪个 worker 已异常。

---

### [ ] T012 清理文档与实际行为不一致的问题

**问题**
- README 中描述的部分行为与实际实现不一致。
- 例如：
  - `DISCORD_WEBHOOK` 文档写为可选，但当前启动逻辑会强制要求
  - 前端技术栈描述含 Tailwind，但项目依赖中并未使用

**整改目标**
- 文档与代码一致，降低部署和接手成本。

**任务内容**
- 更新 README 的启动要求与依赖说明。
- 明确哪些环境变量是必填，哪些是可选。
- 若 `DISCORD_WEBHOOK` 应为可选，则同步修代码；若应为必填，则修文档。
- 更新 Dashboard 技术栈说明。

**涉及文件**
- `README.md`
- `main.go`
- `dashboard/package.json`

**验收标准**
- 新同学仅根据 README 就能正确理解运行条件。
- README 与实际启动行为一致。

## 完成定义

满足以下条件后，可视为本轮生产整改完成：
- 高风险配置接口已经受保护
- 密钥管理不再存在默认密钥回退
- 交易与风控关键路径不再静默失败
- DEX 持仓状态与数据库记录一致
- 核心异常场景具备最小自动化测试覆盖
- README 与真实运行方式一致
